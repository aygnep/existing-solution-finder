我是用一种“**错误信息反向搜工具**”的方法找到的。不是单纯搜“Claude Code 代理”，而是把你的问题拆成几个**独特指纹**去搜：

```text
Claude Code
OpenCode Go
OpenAI-compatible
Anthropic proxy
reasoning_content
thinking block
DeepSeek
```

这类词同时出现的项目很少，所以搜索空间会迅速缩小。比如 `oc-go-cc` 的 README 里就直接写了：它把 Claude Code 的 Anthropic 格式转成 OpenAI Chat Completions，再把 OpenAI 响应转回 Anthropic 格式；并且明确处理 `reasoning_content` ↔ Anthropic `thinking` blocks，这正好命中你的 DeepSeek 报错。([GitHub][1])

---

## 我当时的检索思路

你给出的关键错误是：

```text
The `reasoning_content` in the thinking mode must be passed back to the API.
```

这个错误有几个很强的指向：

第一，`reasoning_content` 是 OpenAI-compatible / DeepSeek / Kimi 一类接口里常见的 reasoning 字段。

第二，Claude Code 原生走的是 Anthropic Messages API，不是 OpenAI Chat Completions。

第三，你用了 OpenCode Go 套餐，而它只支持 OpenAI 格式，所以中间必然需要一个 **Anthropic ↔ OpenAI 协议转换层**。

所以真正该搜的不是：

```text
DeepSeek API error
```

而是：

```text
Claude Code OpenAI compatible Anthropic proxy reasoning_content
```

或者更窄：

```text
Claude Code OpenCode Go reasoning_content GitHub
```

这样就能搜到 `oc-go-cc`、`UniClaudeProxy`、`claude-code-proxy`、`claude-code-router`、`cc-switch` 等一批相关项目。比如 `UniClaudeProxy` 也是 “让 Claude Code 连接 OpenAI-compatible APIs、Gemini、DeepSeek、GLM、Ollama 等后端”的代理项目。([GitHub][2]) 还有 `claude-code-router`，它的文档里也出现了处理 `reasoning_content` 字段的相关配置。([GitHub][3])

## 为什么能找到前几天才创建的项目？

因为新项目虽然没有 SEO 排名，但它们通常会出现在这些地方：

1. GitHub README 关键词非常精确，比如 `Claude Code`、`OpenCode Go`、`reasoning_content`。
2. GitHub issues 会暴露真实错误日志，比如 “missing reasoning_content causes 400 error”。类似问题在 `cc-switch` issue 里也有人遇到。([GitHub][4])
3. 相关项目的 README 会互相提到“related projects”。
4. 搜索引擎对 GitHub 新仓库和 issue 的索引速度很快，尤其是错误字符串很独特的时候。

你这个问题本身就很适合“错误字符串 + 工具生态关键词”检索。

# 如果你想做一个“寻找已有工具解决当前问题”的项目

我建议你不要做成普通搜索引擎，而是做成一个 **Problem-to-Tool Discovery Agent**。

核心输入不是“我想找工具”，而是用户当前的上下文：

```text
我在用 Claude Code + OpenCode Go + DeepSeek，报 reasoning_content 错误
```

系统自动抽取：

```json
{
  "domain": "AI coding agent",
  "software": ["Claude Code", "OpenCode Go", "DeepSeek"],
  "protocols": ["Anthropic Messages API", "OpenAI Chat Completions"],
  "error_terms": ["reasoning_content", "thinking mode"],
  "desired_solution_type": ["proxy", "adapter", "router", "compatibility layer"]
}
```

然后生成多组查询。

---

## 查询生成策略

不要只生成一条 query，要生成 5 类：

### 1. 精确错误搜索

```text
"reasoning_content" "thinking mode" "Claude Code"
```

### 2. 协议桥接搜索

```text
Claude Code OpenAI compatible Anthropic proxy
```

### 3. 生态对象搜索

```text
OpenCode Go Claude Code proxy GitHub
```

### 4. GitHub issue 搜索

```text
site:github.com/issues Claude Code reasoning_content
```

### 5. 替代方案搜索

```text
Claude Code router OpenAI compatible DeepSeek Kimi
```

这样比单搜一个关键词强很多。

# 项目架构可以这样设计

```text
用户问题
  ↓
问题解析器 Problem Parser
  ↓
关键词/错误/软件/协议抽取
  ↓
多查询生成 Query Generator
  ↓
多源搜索 Search Layer
  ├─ GitHub repositories
  ├─ GitHub issues
  ├─ Reddit / HN
  ├─ 官方文档
  └─ package registry
  ↓
候选工具聚类 Candidate Clustering
  ↓
适配度评分 Fit Scoring
  ↓
风险检测 Risk Check
  ↓
推荐结果 + 配置建议
```

---

## 评分标准很关键

不要只按 star 数排序。很多新工具 star 很少，但正好解决问题。

我会设计一个评分：

```text
Fit Score =
  关键词命中度
+ 错误字符串命中度
+ README 中是否有安装/配置
+ 最近维护时间
+ issue 活跃度
+ 是否明确支持用户的栈
+ 是否有示例配置
- 安全风险
- 过期/无人维护
- README 空洞
```

对于你的例子，`oc-go-cc` 虽然很新，但它命中：

```text
Claude Code
OpenCode Go
OpenAI Chat Completions
Anthropic
thinking blocks
reasoning_content
DeepSeek V4
```

所以适配度会非常高。([GitHub][1])

## 数据源优先级

我建议从这些源开始：

Error from provider DeepSeek: reasoning_content in thinking mode must be passed back

这是 Claude/OpenAI reasoning 格式兼容问题

然后输出：
